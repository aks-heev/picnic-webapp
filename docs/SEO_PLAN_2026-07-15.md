**Where it stands:** the least-progressed track relative to how long it's been flagged. **Plan:**
1. **[S to start, then wait] Create the Jaipur GBP profile.** Longest-untouched item in this plan — three docs flag it, none acted on it. Categories: Event planner / Picnic ground, matching Gurugram. Indian GBP verification can take weeks, so start it now regardless of other priorities — the wait dominates the work.
   **Contingency:** if verification stalls >3 weeks or is rejected (common for service-area businesses without storefront signage), fall back to video verification, and prepare evidence beforehand: signage/branded setup photos at a Jaipur venue, a local contact number that answers as "The Picnic Stories."
2. **[M] Enrich the Gurugram profile** per `Google Business Profile Playbook.docx`: service area, additional categories, real hours (not "24h" — reads low-effort to users and Google), photos beyond the single existing one, Services/Products section.
3. **[M] Wire the review ask into `post-event-nudge`** (edge function exists, doesn't send a review link). Day-after-event WhatsApp/email with the per-city GBP review link. Reality check baked in: at ~1 confirmed booking/60 days, this engine produces ~1 review ask per 2 months — it cannot hit any meaningful review count until Track 3 lifts the confirm rate. Build it because it's cheap and compounding, but the review *target* below is explicitly gated on conversion improving. Interim: personally ask the handful of past confirmed guests (there is at least one) for a review — with a base of 0, the first 3 reviews matter more than the engine.
4. **[M] Once real reviews exist, replace the static "4.9 Avg. rating / 200+ Happy couples" trust bar and testimonials section with real sourced content**, and embed 3–4 real review excerpts on venue pages. Shipping fabricated-looking rating claims next to a review engine that produces zero real reviews is an inconsistency worth fixing before scaling traffic into it. If real review volume stays near-zero by day 45, *soften or remove* the fabricated numbers rather than waiting for replacements — unverifiable claims are a downside risk with zero upside.
5. **[S] Citations** (JustDial, Sulekha, WedMeGood or equivalent) — lowest-effort, lowest-priority; sequence after GBP and the review engine, not before.

**Exit condition:** both GBPs live and enriched (or Jaipur in documented verification limbo with evidence prepared); `post-event-nudge` sending review asks; first 3+ real reviews secured (engine + manual asks); homepage trust signals either real or removed — not fabricated.

A dedicated `compute_booking_advance` function already existed at 30%, unused.    - **Not yet done: local repo files need to be committed** (see git block below) — the fix is live in Supabase but the repo's migration file and `notify-booking-received/index.ts` are uncommitted.
   - **Not yet verified: a real end-to-end booking** through the actual UI (this session only verified the RPC directly via SQL) — worth one real test booking + Razorpay checkout to confirm the quoted and charged amounts match to the rupee in the live UI, not just at the database layer.

**Exit condition:** both pricing decisions made and shipped, verified by a test booking whose quoted and charged amounts match; sticky-sidebar bug confirmed fixed or fixed; daily digest live and every abandoner getting followed up with reasons logged; real review data feeding social proof; `/packages` verified on a real device. ## Sequencing across all three tracks

Shared root cause: low, mostly-wasted traffic hitting a leaky funnel with no real social proof. Order that extracts the most from existing traffic before adding more:

1. Pricing decisions (a)+(b) + sticky-sidebar check (Track 3 items 1–2) — stop the leak.
2. Re-submit the 13 GSC URLs; start Jaipur GBP verification (Track 1 item 1, Track 2 item 1) — cheap, additive, run in parallel with step 1. Both are start-and-wait items; the waiting clock should start now.
3. Daily digest + follow-up-with-reason-capture (Track 3 item 3) — convert or at least *understand* the abandoners already arriving before generating more.
4. Jaipur content (Track 1 items 2–3) + Gurugram GBP enrichment (Track 2 item 2).
5. Review engine wiring + manual review asks to past guests (Track 2 item 3).
6. Real social proof everywhere (Track 2 item 4, Track 3 item 4) — gated on actual reviews existing; if none by day 45, remove the fabricated claims instead.
7. Everything else opportunistically (corporate page, FAQs, citations, mobile pass).

## Metrics — with measurement mechanics

**Weekly 15-minute review (Mondays):** (1) GSC Indexing → Pages → "Discovered – currently not indexed" count; (2) GSC Performance filtered to `/blog/` for the post loop; (3) one SQL query against `bookings` for the funnel row (leads / confirmed / abandoned / followed-up-with-reason, last 60d rolling, test phone excluded); (4) GBP dashboards for review count + Jaipur verification status. Log the four numbers in this doc's changelog or PENDING_ITEMS. If a number moves against target for 2 consecutive weeks, that track's contingency triggers.

| Metric | Baseline (today) | 60-day target | Measured via |
|---|---|---|---|
| GSC "Discovered — not indexed" count | 13 | 0 | GSC Indexing report, weekly |
| Confirmed bookings / real leads (60d) | 1 / ~6–7 | 3+ / same or higher | SQL on `bookings`, weekly |
| Abandoned leads followed up w/ reason logged | 0% | 100% of new abandons | SQL: notes present on abandoned rows |
| Abandoned-lead rate | ~71% (5/7) — *directional, n=7* | <50% — *directional* | SQL, read as trend not threshold |
| GBP profiles live & enriched | 1 thin, 1 missing | 2 enriched (or Jaipur in documented verification) | GBP dashboards |
| Real reviews (both cities) | 0 | 3+ (floor via manual asks); 5–10 only if confirm rate rises | GBP review counts |
| Jaipur occasion content pieces | 0 | 3 (2 posts + listicle) | ship log |
| Blog posts w/ impressions >0 in GSC | unknown (pull at week 1) | all live posts | GSC Performance, `/blog/` filter |

Targets on funnel metrics are **directional** until lead volume is meaningfully above ~10/60d — treat 2-week trends and logged abandonment reasons as the real signal, not the percentages.
